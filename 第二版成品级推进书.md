# 《余光未熄》第二版推进书

## 1. 文档定位

本文档用于推进《余光未熄》的第二版开发。

第二版不是最终完整版，也不是商业级最终成品。第二版是从“功能原型”走向“完整可演示版本”的中间阶段，目标是把一局游戏的入口、UI、玩法闭环、网页展示和后续扩展基础打稳。

最终成品仍然会继续向以下方向推进：

- 真正多人联机：每个玩家只操控自己的一名角色。
- 更完整的角色、地图、动画、音效和美术资源。
- 更复杂的光影机制和阵营对抗。
- 更完整的战绩、设置、测试和课程报告。
- 更高完成度的网页部署和课堂展示流程。

因此，本推进书的重点不是一次性完成最终游戏，而是明确第二版当前已经完成什么、还差什么、下一步如何继续做。

## 2. 当前项目总览

当前项目已经从早期测试原型推进到 Raylib C++ 图形化单局演示，并已经支持 WebAssembly 网页版本。

当前运行方式有两种：

```text
桌面版：start_game.ps1
网页版：start_web.bat / start_web.ps1
```

目前推荐展示方式是网页版：

```text
双击 start_web.bat
打开 http://localhost:8080
```

当前 Web 版已经实现：

- 本地网页服务器启动。
- 浏览器自动打开。
- 游戏画面填满整个网页。
- 中文字体资源打包进 Web 版本。
- Raylib C++ 通过 Emscripten 编译为 HTML / JS / WASM。

当前桌面版仍保留，主要用于本地调试和 smoke 测试。

需要特别说明：当前版本里“一台电脑控制 3 名人类和 1 名影”只是第二版的本地演示 / 调试方式，不是最终成品玩法。最终成品应是多人联机房间制，每名玩家进入房间后选择或分配一名角色，并且整局只操控该角色。

### 3.4 当前玩法

当前已经能跑通一局本地演示。

当前本地演示为了方便测试，允许同一操作者切换控制多名人类，并同时控制影。这个设计用于验证规则闭环、任务链和 UI，不代表最终操作方式。

人类阵营已有：

- 3 名人类角色。
- `Q / E` 切换当前人类。
- `W / A / S / D` 移动。
- `H` 点亮灯源。
- `J` 在任务点协作。
- `K` 拾取或交付光核。
- `L` 救援 / 净化附近队友。

影阵营已有：

- 1 名影角色。
- 方向键移动。
- 可穿墙移动。
- `1 - 6` 房间传送。
- 被强光照射后进入虚弱状态。

光影机制已有：

- 手电扇形光照。
- 灯源安全光区。
- 黑暗遮罩。
- 人类有限视野 / 影全图视野切换。
- 影被强光压制。
- 影靠近人类会推动同化。

任务链已有：

- 三灯校准。
- 双闸供能。
- 运送光核。
- 最终净化。

胜负条件已有：

- 光阵营完成最终净化获胜。
- 所有人类被同化时影阵营获胜。
- 当前第二版不设置回合上限，影阵营需要通过同化全部人类获胜。

## 4. 第二版目标重新定义

第二版的正确目标是：

```text
完成一个可双击启动、可在网页展示、可完整演示一局、UI 基本完整、后续能继续扩展联机和美术资源的阶段版本。
```

第二版允许继续使用“本地多人模拟”来验证规则，但文档、UI 和代码结构都要朝最终成品方向靠拢：最终成品中每个玩家只拥有一个 `PlayerId` 和一个被分配的角色，不能像当前测试版一样由一个人随意切换操控全部角色。

第二版不是最终成品，因此不要求现在一次性完成：

- 真正网络联机。
- 完整商业级美术。
- 完整角色动画精灵表。
- 完整音效和 BGM。
- 完整战绩平台。
- 多地图和多关卡。
- 完整 AI 导演。

第二版中诸如最终净化条件、任务提示细节、UI 字号、地图房间布局等，都属于后续可以继续调的内容。只要当前功能闭环能跑通，就可以把它视为第二版功能阶段成果。

第二版必须保证的是：

- 老师或同学能打开。
- 能看到完整入口。
- 能看懂当前目标。
- 能操作人类和影。
- 能完成或阻止任务。
- 能进入结算。
- 能说明后续如何扩展到最终版。

## 5. 第二版剩余任务

### 5.1 P0：当前第二版必须继续补齐

这些是第二版进入稳定演示前最优先完成的内容。

#### 5.1.1 游戏内反馈增强

当前问题：

- 任务完成反馈还不够明显。
- 点灯、协作、救援、同化等事件缺少明显提示。
- 玩家有时不知道“刚才操作有没有成功”。

需要完成：

- 增加任务完成提示。
- 增加点灯成功提示。
- 增加救援成功提示。
- 增加光核拾取 / 交付提示。
- 增加影虚弱提示。
- 增加同化上升提示。

验收标准：

- 不看代码也能知道每个关键操作是否成功。
- 任务推进时 UI 有明确反馈。

#### 5.1.2 对局准备界面继续打磨

当前问题：

- 现在的光阵营 / 影阵营选择更多是“视角选择”，还不是真联机房间。
- 准备确认逻辑是本地自动准备。
- 当前本地演示会让一个操作者控制多个角色，容易被误解为最终玩法。

需要完成：

- 明确写成“本地联机演示模式”。
- UI 文案避免让人误以为已经有网络联机。
- 增加“后续联机预留”的说明或结构。
- 增加最终成品说明：真实联机时每名玩家只操控一名角色。

验收标准：

- 老师看界面时不会误解为已经完成真实网络联机。
- 老师能看懂当前“一人控制多角色”只是演示和测试模式。
- 后续接入网络时可以复用该入口流程。

#### 5.1.3 操作说明继续完善

当前问题：

### 5.3 P2：最终成品方向预留

这些不要求第二版全部完成，但要在结构上预留。

#### 5.3.1 真正联机

最终版的核心玩法必须改为：

```text
一个房间内有多名玩家，每名玩家只操控一名角色。
```

推荐目标形态：

- 玩家 A 控制守灯人。
- 玩家 B 控制工程师。
- 玩家 C 控制观测员。
- 玩家 D 控制影。

如果人数不足，可以后续再考虑 AI 托管或本地补位，但最终成品的默认设计不应是一个人控制全部角色。

最终版需要：

- 房间创建。
- 加入房间。
- 玩家准备。
- 阵营分配。
- 角色选择或角色分配。
- 每个玩家绑定一个角色。
- 每个客户端只接收自己角色的输入。
- 玩家输入同步。
- 角色状态同步。
- 任务状态同步。
- 胜负结果同步。

第二版只需要做到：

- 游戏状态集中在 `Game` 中。
- 角色、任务、灯源状态结构清晰。
- UI 不直接散乱修改底层数据。
- 开始预留 `PlayerId -> CharacterId` 的思路。
- 当前本地演示保留多角色控制，但后续要能拆成单玩家单角色输入。

#### 5.3.2 更完整美术

最终版需要：

- 角色精灵表。
- 地图背景。
- 灯源、光核、封印台等物件。
- 手电、同化、净化等特效。
- 主菜单和结算背景。

第二版只需要做到：

- 素材目录清晰。
- 图形占位能稳定运行。
- 后续素材可以逐步替换。

第三版的重点应放在素材导入和视觉表现升级：把角色、地图、灯源、光核、任务点、UI、图标、音效等素材逐步导入当前框架，替换第二版中的几何占位图。

地图设计不需要在第二版定死。第三版导入地图素材时，可以继续优化：

- 房间尺寸和通道宽度。
- 任务点位置。
- 灯源位置。
- 影的传送点。
- 人类初始点。
- 封印大厅和终局区域。
- 地图整体观感和辨识度。

只要核心任务链不被破坏，地图完全可以在第三版继续调整。

#### 5.3.3 多局和战绩展示

最终版需要：

- 历史对局列表。
- 阵营胜率。
- 玩家表现统计。
- 可视化结算。

第二版只需要做到：

- 有基础战绩文件。
- 能写入、读取、更新。
- 报告能说明文件处理逻辑。

- 网络房间。

- 准备状态同步。
- 阵营分配同步。
- 每名玩家选择或分配一名角色。
- 每名玩家只能操控自己的角色。
- 人类玩家之间通过各自角色协作完成任务。

- 输入同步。

## 9. 推荐下一步

下一步不要急着做真正联机。当前最合适的顺序是：

1. 先做操作反馈和结算增强。
2. 再做战绩记录系统，替代中途存档。
3. 再预留玩家槽位和“每名玩家绑定一名角色”的数据结构。
4. 再做素材加载接口，为第三版批量导入素材做准备。
5. 第三版开始导入角色、地图、UI、特效、音效等正式素材。
6. 再补音效和简单动画。
7. 最后整理测试报告和最终报告。

这样推进的原因：

- 当前已经能打开网页，也能跑一局。
- 现在最缺的是“看起来像完整版本”的反馈和结算。
- 文件处理要求可以用战绩记录解决，不需要做偏离联机方向的存档。
- 最终成品是多人联机单角色操控，所以要尽早预留玩家槽位结构。
- 真正联机难度较高，应该等单局本地版本稳定后再接入。

## 10. 当前结论

当前项目已经不是最早的测试原型，而是一个可网页打开、可演示入口和基础玩法的第二版雏形。

但第二版仍未完全收口，尤其缺少：

- 操作反馈。
- 结算细节。
- 战绩记录。
- 音效。
- 素材接入。
- 测试报告。

最终成品距离现在还有明显距离。当前阶段应该继续稳步推进，不要把第二版误认为最终版。第二版的作用是打牢“完整单局演示 + 网页展示 + 后续联机扩展”的基础。

最终成品的核心方向必须保持清楚：不是一个人操控三名人类和影，而是多人进入同一局，每名玩家操控一名角色，通过协作或对抗完成一整局。
